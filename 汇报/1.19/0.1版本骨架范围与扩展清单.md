# 0.1版本最小可用骨架范围与后续扩展清单

## 一、0.1版本最小可用骨架的范围界定

### 1.1 核心目标
**0.1版本的目标：** 实现一个最简化的"需求 → 经验路径匹配 → 函数级逻辑方案生成"流程，能够：
- 接收自然语言需求描述
- 从已有项目的功能分区/路径中找到最相似的参考路径
- 生成一个"新功能应该按什么函数调用链来组织"的方案
- 通过API返回方案，供前端展示或人工查看

**不包含的内容（留给后续版本）：**
- ❌ 不生成完整代码实现
- ❌ 不做代码级依赖检查和冲突检测
- ❌ 不支持多项目/跨库匹配
- ❌ 不做方案自动优化和迭代

---

### 1.2 最小功能模块（3个核心类 + 1个API路由）

#### 模块1：`RequirementInterpreter`（需求解析器）
**位置：** `llm/requirement_interpreter.py`

**职责：**
- 接收自然语言需求描述（字符串）
- 调用LLM将需求解析成结构化信息（功能关键词、输入输出类型、涉及的功能域）
- 返回标准化的需求对象

**最小实现范围：**
```python
class Requirement:
    description: str           # 原始需求描述
    keywords: List[str]        # 提取的关键词（如["认证", "登录"]）
    input_types: List[str]     # 输入类型推断（如["用户名", "密码"]）
    output_types: List[str]    # 输出类型推断（如["令牌"]）
    domain: str               # 功能域（如"用户认证"）

class RequirementInterpreter:
    def interpret(self, requirement_text: str) -> Requirement
```

**依赖：**
- 复用现有的 `CodeUnderstandingAgent` 中的LLM配置
- 调用DeepSeek API（同现有系统）

**输出示例：**
```json
{
  "description": "添加用户OAuth登录功能",
  "keywords": ["oauth", "登录", "认证", "第三方"],
  "input_types": ["授权码", "回调URL"],
  "output_types": ["访问令牌", "用户信息"],
  "domain": "用户认证"
}
```

---

#### 模块2：`ExperiencePathMatcher`（经验路径匹配器）
**位置：** `analysis/experience_path_matcher.py`

**职责：**
- 从已分析的项目中加载功能分区、路径、CFG/DFG摘要
- 根据需求的关键词和功能域，在经验库中匹配最相似的路径
- 返回Top-K候选路径（每条路径包含：函数调用链、I/O摘要、所在分区）

**最小实现范围：**
```python
class ExperiencePath:
    path_id: str                    # 路径ID（如"partition_0_path_3"）
    partition_id: str               # 所在分区ID
    function_chain: List[str]       # 函数调用链（如["validate", "query_db", "generate_token"]）
    io_summary: Dict[str, Any]      # I/O摘要（输入类型、输出类型）
    semantic_label: str             # 语义标签（如"用户登录流程"）

class ExperiencePathMatcher:
    def __init__(self, project_path: str):
        # 从缓存/分析结果中加载分区和路径数据
    
    def match(self, requirement: Requirement, top_k: int = 3) -> List[ExperiencePath]
```

**数据来源：**
- 复用 `function_hierarchy_cache`（`app.py`中已有）
- 从 `partition_analyses[partition_id]['path_analyses']` 中提取路径信息
- 可选：用 `MethodFunctionProfileBuilder` 为路径生成语义标签

**匹配策略（0.1版本简化版）：**
1. **关键词匹配：** 在路径的函数名、分区名、语义标签中查找需求关键词
2. **功能域匹配：** 如果需求能推断出功能域，优先匹配该域的分区
3. **LLM辅助匹配（可选）：** 如果关键词匹配结果太少，调用LLM做语义相似度判断

**输出示例：**
```json
{
  "matched_paths": [
    {
      "path_id": "partition_0_path_3",
      "partition_id": "partition_0",
      "function_chain": ["UserService.validate_input", "UserService.query_database", "TokenService.generate_token"],
      "io_summary": {"input": "用户名+密码", "output": "令牌+用户信息"},
      "semantic_label": "用户登录流程",
      "similarity_score": 0.85
    },
    ...
  ]
}
```

#### 模块3：`FunctionFlowPlanner`（函数级逻辑方案生成器）
**位置：** `llm/function_flow_planner.py`

**职责：**
- 根据需求描述和匹配到的参考路径，生成新功能的函数调用逻辑方案
- 方案包括：步骤列表、每步调用的函数、哪些函数需复用、哪些需新增

**最小实现范围：**
```python
class PlannedStep:
    step_index: int                 # 步骤序号
    description: str                # 步骤描述（如"验证用户输入"）
    function_calls: List[str]       # 调用的函数（可以是现有函数或新函数占位符）
    input_vars: List[str]           # 输入变量
    output_vars: List[str]          # 输出变量

class PlannedFunctionFlow:
    requirement: Requirement        # 原始需求
    reference_paths: List[ExperiencePath]  # 参考路径
    steps: List[PlannedStep]       # 函数调用步骤列表
    new_functions: List[Dict]      # 需要新增的函数（名称、模块、输入输出建议）

class FunctionFlowPlanner:
    def plan(self, requirement: Requirement, reference_paths: List[ExperiencePath]) -> PlannedFunctionFlow
```

**生成逻辑（0.1版本）：**
1. **LLM生成：** 将需求 + 参考路径摘要喂给LLM，生成步骤列表
2. **函数映射：** LLM需要判断：哪些步骤可以直接复用现有函数，哪些需要新函数
3. **命名规范：** 新函数的命名要符合项目风格（从参考路径中学习）

**输出示例：**
```json
{
  "requirement": {...},
  "reference_paths": [...],
  "steps": [
    {
      "step_index": 1,
      "description": "验证OAuth授权码",
      "function_calls": ["OAuthService.validate_auth_code"],
      "input_vars": ["auth_code", "redirect_uri"],
      "output_vars": ["validated_code"]
    },
    {
      "step_index": 2,
      "description": "调用第三方OAuth服务获取令牌",
      "function_calls": ["OAuthService.exchange_token"],
      "input_vars": ["validated_code"],
      "output_vars": ["access_token"]
    },
    ...
  ],
  "new_functions": [
    {
      "name": "OAuthService.validate_auth_code",
      "module": "services/oauth_service.py",
      "suggested_signature": "validate_auth_code(auth_code: str, redirect_uri: str) -> dict",
      "description": "验证OAuth授权码的有效性"
    },
    ...
  ]
}
```

#### 模块4：后端API路由
**位置：** `app.py` 新增路由

**API接口：**
```python
@app.route('/api/requirement/plan', methods=['POST'])
def api_requirement_plan():
    """
    根据需求生成函数级逻辑方案
    
    请求体：
    {
        "requirement": "添加用户OAuth登录功能",
        "project_path": "D:/code/my_project"  # 可选，默认使用当前项目
    }
    
    返回：
    {
        "requirement": {...},           # 解析后的需求
        "matched_paths": [...],         # 匹配到的参考路径
        "planned_flow": {...},          # 生成的函数调用方案
        "status": "success"
    }
    """
```

**最小实现：**
- 接收POST请求，解析需求文本
- 依次调用：`RequirementInterpreter` → `ExperiencePathMatcher` → `FunctionFlowPlanner`
- 返回JSON结果

---

### 1.3 数据流图（0.1版本）

```
用户输入需求文本
    ↓
RequirementInterpreter (LLM解析)
    ↓
结构化需求对象
    ↓
ExperiencePathMatcher (关键词/功能域匹配)
    ↓
候选参考路径列表 (Top-K)
    ↓
FunctionFlowPlanner (LLM生成方案)
    ↓
函数级逻辑方案 (JSON)
    ↓
API返回给前端/用户
```

## 二、后续扩展功能清单（按优先级排序）

### 2.1 短期扩展（0.2-0.3版本）

#### 扩展1：路径语义画像完善
**目标：** 为每条路径生成更丰富的语义标签和功能描述

**需要做的事：**
- 完善 `MethodFunctionProfileBuilder`，确保所有路径都有语义画像
- 为路径生成标准化的功能标签（如"用户认证"、"数据处理"、"API路由"）
- 建立路径画像索引（JSON文件或简单数据库），支持快速检索

**预计工作量：** 2-3天

---

#### 扩展2：LLM增强的路径匹配
**目标：** 用LLM做语义相似度匹配，而不仅依赖关键词

**需要做的事：**
- 在 `ExperiencePathMatcher` 中增加LLM语义匹配模块
- 将需求描述和路径摘要同时喂给LLM，让LLM判断相似度
- 缓存匹配结果，避免重复调用

**预计工作量：** 1-2天

---

#### 扩展3：方案验证与依赖检查
**目标：** 检查生成的方案是否违反项目依赖规则

**需要做的事：**
- 新增 `PlanValidator` 类，检查：
  - 新函数是否与现有函数重名
  - 调用的现有函数是否存在
  - 模块依赖关系是否合理（避免循环依赖）
  - 输入输出类型是否匹配
- 在 `FunctionFlowPlanner` 生成方案后，自动调用验证器

**预计工作量：** 2-3天

---

#### 扩展4：前端可视化页面
**目标：** 为"新功能方案"提供可视化界面

**需要做的事：**
- 新增前端页面（如 `templates/requirement_plan.html`）
- 复用现有的Cytoscape.js图可视化组件
- 展示：需求解析结果、匹配路径、生成的函数调用流程图
- 允许用户修改方案并重新生成

**预计工作量：** 3-4天

## 五、总结

### 0.1版本的核心价值
- **最小可用：** 只需3个核心类 + 1个API，代码量约650行
- **立即可用：** 能完成"需求 → 匹配 → 方案生成"的完整流程
- **易于扩展：** 架构清晰，后续功能可以逐步叠加

### 后续扩展的规划思路
- **短期（0.2-0.3）：** 提升匹配准确度、增加验证、完善前端
- **中期（0.4-0.5）：** 支持迭代优化、多方案对比、详细签名生成
- **长期（0.6+）：** 代码生成、跨项目复用、增量学习等高级功能

### 风险点
1. **LLM调用成本：** 每个需求可能需要多次LLM调用，需要做好缓存和优化
2. **匹配准确度：** 如果项目规模大、路径多，关键词匹配可能不够准确，需要尽快加入LLM语义匹配
3. **数据质量依赖：** 方案的生成质量高度依赖已有分析的路径数据质量，需要确保功能分区分析足够完善
