## 老师设计文字与解释（能力从“学代码”到“用代码”）

### 一、老师关于“人机协同、让新人写出与老人同风格代码”的目标

> **老师原话（节选概括自图片 1）：**  
> 您的问题描述对应文档中“人类明确需求、多智能体协同执行”的人机协同目标。具体来说：  
>  
> **输入：**  
> - 一个现有代码库（如 GitHub 项目），系统已通过知识解析阶段将其转化为函数知识图谱（捕获函数节点、调用关系、依赖结构）。  
> - 一个需求功能描述（如“添加用户认证功能”或“迁移到新框架”），可以是一句话的模糊想法，也可以是一整页的复杂需求。  
>  
> **输出：**  
> - 一个新的代码逻辑架构（即实现该需求的代码组织方案），该架构会尽可能符合原代码库的结构风格，确保与现有代码的兼容性和一致性。  
>  
> 这一过程旨在解决文档中强调的“资源内耗”和“协同失效”痛点——通过重用现有代码库的知识和经验，避免重复开发，降低集成风险。

**解释：**  

- **输入端**的关键是：系统已经把“老项目的代码”学成了一个函数级知识图谱，里面有谁调谁、依赖关系、模块边界等；再加上“人类用自然语言给的新需求描述”。  
- **输出端**并不是直接生成完整代码，而是先生成一个“代码逻辑架构 / 函数调用方案”——相当于一个高级别的“设计稿 + 调用链规划”，告诉新人：  
  - 该复用哪些现有函数 / 模块；  
  - 新功能大概要分成哪些步骤；  
  - 每一步应该放在哪个模块、以什么风格命名、按什么调用顺序来组织。  
- 这样，新人写出来的实现，在结构、函数拆分方式、调用逻辑上，都尽量“像老员工写的”，从而减少“推倒重来”和不必要的 API / 函数重复造轮子。

### 二、老师关于“三层架构：知识→经验→能力”的技术路线

#### 1. 知识解析阶段（代码 → 知识）

> **老师原话（图片 2 中第 1 点）：**  
> 系统通过程序静态分析（如 AST 解析、依赖分析）将原代码库转化为函数知识图谱。该图谱完整捕获了代码的层级结构（文件-包-类-函数）和调用关系（如函数 A 调用函数 B）。  
>  
> **关键作用：**  
> ==图谱成为“架构蓝图”，定义了原代码库的设计模式、接口规范和依赖规则。任何新功能都必须遵循这些固有模式。==

**解释：**  
- 这一步基本就是你现有系统已经在做的事情：解析 Python 项目，得到类、函数、调用关系、跨文件依赖、数据流等，然后构建知识图谱 / 调用图 / 超图、CFG、DFG。  
- 它回答的是：**“老项目现在是怎么写的？”**——包括结构、调用关系、控制流、数据流。  
- 在老师的路线图里，这一层叫“知识层”，是后面“经验”和“能力”的原始材料。

#### 2. 经验沉淀阶段（知识 → 经验）

> **老师原话（图片 2 中第 2 点）：**  
> 系统从知识图谱中提取**任务经验图谱**，即常见功能路径（例如“用户登录”功能可能涉及“验证输入 → 查询数据库 → 生成令牌”的调用链）。  
>  
> **关键作用：**  
> ==任何经验图谱封装了原代码库的“最佳实践”。当处理新需求时，系统会优先匹配这些已知路径，确保新功能与原架构风格一致。==

**解释：**  

- ==在知识层只是“知道所有可能的调用关系”；经验层则是**挑出典型路径 / 模式**，把它们当作可复用的“案例库”==：  
  - 某一类功能通常由哪几个函数串起来；  
  - ==中间会经过哪些关键检查、错误处理、日志记录==  ==（这里其实现在的项目不一定能做到，项目只会观察主干功能，并没有测试日志 错误处理，关键检查这些后续需要可以加上）==
  - 输入 / 输出在路径上的演化方式。  
- 你项目里已经有“功能分区（社区检测）”“函数调用超图 Hypergraph”“路径级 CFG/DFG”“Path-level 分析器”等模块，这些其实就是在往“经验层”靠：  
  - 把一大坨函数图拆成“功能分区”；  
  - 再把分区里的调用图拆成多条具体路径（子功能）；  
  - 每条路径再配上 CFG/DFG 和 I/O 信息。  
- 老师强调的“经验沉淀”，就是要把这些**路径 + 语义 + I/O**整理成可检索、可匹配的新需求“参考模板”。

#### 3. 能力转化阶段（经验 → 能力）

> **老师原话（图片 2 红框部分）：**  
> **项目专家智能体**接收需求后，通过“规划-执行-反思”机制：  
>
> - **规划**：解析需求，在任务经验图谱中匹配相似的功能路径（如需求是“增加 OAuth 登录”，则匹配原代码库的“用户认证”相关路径）。  
> - **执行**：基于匹配路径，调用知识图谱中的现有函数节点，生成新的代码方案。系统会确保新代码的输入输出、错误处理与原架构一致（如复用相同的接口规范）。  
> - **反思**：校验新架构是否违反原代码的依赖规则或设计约束（如避免循环依赖）。  
>
> **全局专家智能体（针对跨项目需求）**：协调多个代码库，确保新架构在跨库调用时仍符合整体逻辑一致性。

**解释：**  
- 这一层才是真正实现“新人写出老人风格代码”的关键：  
  - **项目专家智能体**相当于“这个仓库里的老大哥”，熟悉本仓库所有常见功能实现套路；  
  - 当你给一个新需求描述时，它会先在“经验图谱”里找：**有没有类似功能已经被老代码实现过？**  
  - 找到后，它会**仿照老的调用链**，给出一个“新需求的函数级逻辑方案”：建议复用哪些旧函数、在哪一步插入新的处理、哪些地方需要新函数但应按什么命名规范、放在哪个模块。  
- **规划-执行-反思**本质是一个闭环：  
  - 规划：从经验库中选“原型”；  
  - 执行：基于原型生成新的逻辑草图（函数调用流程 / 超图）；  
  - 反思：用知识图谱 / 规则检查兼容性（依赖、层次划分、输入输出一致性）。  
- 对于更大范围（多个仓库）的需求，还需要一个“全局专家智能体”去平衡不同项目之间的风格和接口，但这是更远一步的扩展。

### 三、整体汇总理解（把老师的愿景用一句话说清楚）

**整体来看，老师希望构建的是：一个先“把老项目学懂”，再“用老项目的写法来规划新功能代码”的智能助手。**  

- 现在的系统已经基本完成了“**学懂老项目**”这一半：  
  - 能建立函数级知识图谱（调用关系、继承、跨文件依赖、数据流）；  
  - 能做功能分区、路径级超图、CFG/DFG 和 I/O 分析；  
  - 有 LLM Agent（`CodeUnderstandingAgent`、`GraphKnowledgeBase` 等）来理解项目结构和重要代码。  
- 下一步要做的是“**把这些理解转化成对新需求的行动能力**”：  
  - 输入：新功能需求的自然语言描述；  
  - 过程：在“功能分区 + 超图 + 路径经验”里查找最相近的实现套路，再仿照这些路径，规划一个新功能的函数级逻辑流程；  
  - 输出：一份“函数调用逻辑 / 超图方案”（或者更进一步的伪代码草图），风格和结构高度贴近原项目，让新人照着走就能写出与老人同逻辑的实现。  
- 换句话说：**老师要的是“用知识图谱 + 超图，把老项目的隐性经验变成可复用的设计模版”，再用智能体把这些模版应用到新需求上。**

### 四、当前代码对老师目标的支持情况与不足

#### 4.1 已经具备的基础能力（基本覆盖“知识解析 + 部分经验沉淀”）

- **代码解析与知识图谱构建已经较完善**：  
  - `analysis/analyzer.py` 负责解析项目、构建类 / 函数 / 调用关系，并生成图数据。  
  - `analysis/call_graph_analyzer.py`、`cross_file_analyzer.py`、`data_flow_analyzer.py` 等实现了调用图、跨文件依赖、数据流分析。  
  - `app.py` 中的 `build_knowledge_graph_data` 会把这些信息组织成更丰富的知识图谱（包含 CFG、DFG、参数、返回值等实体）。  
- **功能分区与函数超图**：  
  - `analyze_function_hierarchy` + `CommunityDetector` 基于调用图做社区检测，得到功能分区（对应“功能层级分析 / 超图提取”）。  
  - `analysis/function_call_hypergraph.py` 构建函数调用超图，并识别路径模式、循环等，为“经验路径”打了基础。  
  - `analysis/path_level_analyzer.py` 和相关 CFG/DFG 生成器，为每条路径生成控制流 / 数据流图和 I/O 信息。  
- **LLM 辅助理解项目结构**：  
  - `llm/code_understanding_agent.py` 已经实现了：阅读 README、构建项目结构摘要、识别重要代码、结合符号表和图知识库摘要来理解项目主干逻辑。  
  - `llm/graph_knowledge_base.py` 能对图数据做压缩总结，用于给 LLM 提供“图知识摘要”。

这些能力基本完成了老师路线中的前两步：**代码→知识→部分经验**。

#### 4.2 针对老师“根据新需求生成函数级逻辑方案”的不足之处

- **缺少“需求输入端”和需求解析模块**：  
  - 目前没有一个统一的入口（API 或类）来接收“自然语言新需求”，也没有专门的 Prompt / 结构把“需求 + 经验图谱摘要”一起喂给 LLM。  
  - 现有的 LLM 调用更多是“解释现有代码 / 划分功能分区”，而不是“基于新需求规划新的代码逻辑”。
- **经验图谱还停留在“分析结果”，没有整理成可检索的经验库**：  
  - 虽然有功能分区、超图和路径级 CFG/DFG，但这些结果主要用于可视化与理解，还没有：  
    - 把每条路径整理成“功能模板 + 语义标签 + I/O 签名”；  
    - 建立一个“路径经验索引”，支持按需求描述进行相似度检索。  
  - `analysis/method_function_profile_builder.py` 等“方法功能画像”相关代码目前还不完整，需要先补齐，才能真正形成“函数级语义画像库”。
- **缺少“规划-执行-反思”的新需求工作流**：  
  - 还没有一个类似“`RequirementPlanningAgent`”的模块，负责：  
    - 读取自然语言需求；  
    - 在经验库中匹配相似功能路径；  
    - 规划出新功能的函数调用链 / 超图；  
    - 用已有的知识图谱规则检查新方案是否违背依赖 / 分层约束。  
  - 也没有对应的 API 路由和前端展示页面来显示“新功能的逻辑架构图”（例如一张新的函数超图或路径列表）。
- **尚未定义“输出格式标准”**：  
  - 老师希望输出是“新的代码逻辑架构 / 函数级流程”，但目前项目没有统一的数据结构来表达：  
    - 一个“新功能设计方案”具体长什么样（例如：步骤列表 + 对应函数 + 输入输出约束 + 需要新增的函数草图）。  
  - 需要先设计好这样的数据模型，LLM 才能有明确的生成目标。

#### 4.3 建议的前置任务清单（0.1 版本可落地的方向）

- **任务 1：补齐并打通“函数/路径语义画像”链路**  
  - 完善 `MethodFunctionProfileBuilder`，为每个方法 / 路径生成：功能描述、输入输出要点、关键调用、所在功能分区等画像，并落地成可查询的 JSON / 索引文件。  
  - 将路径级 CFG/DFG + I/O 信息整理成“经验路径记录”，形成最小可用的“任务经验图谱”。
- **任务 2：设计“需求 → 经验路径匹配”的 LLM Prompt 与接口**  
  - 新增一个 `RequirementInterpreter`（可以是独立类或挂在 `CodeUnderstandingAgent` 上）：  
    - 输入：自然语言需求 + 项目 / 分区 / 路径经验摘要；  
    - 输出：若干“候选经验路径 + 匹配理由”。  
  - 先不生成代码，只生成“应该参考哪些现有路径”的决策结果。
- **任务 3：定义“新功能逻辑方案”的数据结构，并实现生成逻辑**  
  - 设计一个 `PlannedFunctionFlow` / `PlannedHypergraph` 结构，用来描述：  
    - 该功能分成哪几个步骤；  
    - 每步调用哪些现有函数；  
    - 哪些地方需要新函数（以及建议名称、所在模块、输入输出）。  
  - 在此基础上，用 LLM（结合知识图谱约束）生成 0.1 版的“函数级逻辑过程”，不直接落地成代码实现。
- **任务 4：增加一个简单的后端 API 与可视化页面**  
  - 后端增加 `POST /api/requirement/plan` 之类的接口：接收需求描述，返回规划好的函数调用流程。  
  - 前端可以先复用现有的图可视化组件，把“新功能方案”以路径 / 子图的形式展示出来，方便人工检查和迭代。

完成以上前置任务后，你就能有一个**0.1 版本**：  
- 人类给一句或一段需求；  
- 系统基于已有知识图谱 + 路径经验，给出一个“和当前仓库风格高度一致的函数级逻辑方案”；  
- 新人照着这个方案写代码，就能大幅减少“重新构思逻辑”和“乱写 API”的成本，这与老师设想的场景是对齐的。
