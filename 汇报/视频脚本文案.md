## 【开场 0:00–0:30】

**【画面】** 封面：项目名称 + 一句话价值主张

**【旁白】**

大家好。今天要介绍的是一个**智能代码理解与生成平台**。

它的核心价值只有一句话：**让大型代码仓库从「读不完、理不清」变成「可问、可查、可生成」**。

接下来，我会从三个部分展开：**AI学**、**AI用** 和 **AI写**，分别对应代码分析、RAG 问答和代码生成。每一部分都直指开发者在面对大型项目时的真实痛点。

---

## 【痛点引入 0:30–1:30】

**【画面】** 列出 5 个痛点，可配图：开发者面对满屏代码、文档缺失

**【旁白】**

先说说我们想解决的痛点。

接手一个几万行、甚至几十万行的项目时，开发者常会遇到这些问题：

**第一，读不完。** 手动通读代码，动辄要几周，还不一定能形成清晰的整体认知。

**第二，理不清。** 类、方法、模块之间调用关系复杂，很难在脑中建立一张清晰的「地图」。

**第三，找不到。** 想知道「某个功能是怎么实现的」，往往不知道从哪个文件、哪个方法开始看。

**第四，问不了。** 代码不像文档，不能直接检索、不能直接提问，只能靠记忆和搜索关键词。

**第五，改不动。** 理解成本高，改代码时担心牵一发动全身，不敢轻易动手。

这些问题，本质上是：**代码没有被打造成结构化的知识**，只能靠人肉去「学」。

我们的思路是：先让 AI 把代码「学」好，再让 AI 来「用」和「写」。

---

## 【第一部分：AI学 1:30–4:00】

**【画面】** 切换到「AI学」板块，展示分析 pipeline 示意图

**【旁白】**

第一部分，**AI学**：代码分析系统。

我们要做的，不是简单统计行数或画一个文件树，而是**围绕「方法」这个核心单元**，构建多维度的代码知识。

为什么是方法？因为人理解代码时，最自然的粒度就是「这个函数/方法是干什么的」「它调用了谁」「数据从哪来、到哪去」。文件、目录，更多是物理组织，而不是认知组织。

---

**【画面】** 展示：调用图、CFG、DFG、超图、功能分区 的示意

**【旁白】**

因此，我们构建了多种图：

- **调用图**：方法之间的调用关系，一目了然谁调用了谁；
- **控制流图 CFG**：方法内部的分支、循环、条件，逻辑结构清晰可见；
- **数据流图 DFG**：变量的读写和流动，输入输出、副作用都能追踪；
- **超图**：处理一对多、多对多的复杂调用模式；
- **功能分区**：通过 LLM 和社区检测，把方法按「功能模块」聚合，比如核心业务、工具辅助、外部接口。

此外，还有**经验路径**——从入口点到关键逻辑的调用链，带有 CFG、DFG 和语义描述，便于按「功能路径」去理解代码。

整体上，我们采用一个**四层嵌套模型**：功能层、文件夹层、代码元素层、代码细节层。这样，既能从宏观看模块划分，又能从微观看具体实现。

---

**【画面】** 展示可视化界面：交互式图谱、节点类型、关系

**【旁白】**

所有分析结果，都会通过**交互式可视化**呈现。你可以缩放、拖拽、搜索，点击节点查看详情。支持分区级、路径级的不同视图。

一句话总结 AI学：**把人读代码的认知过程，转化为自动化的分析与图谱，让人用「功能」的视角理解代码。**

---

## 【第二部分：AI用 4:00–5:30】

**【画面】** 切换到「AI用」板块，展示 RAG 问答流程

**【旁白】**

第二部分，**AI用**：RAG 问答系统。

有了结构化的代码知识，下一步就是让它**可问**。

用户可以像问文档一样问代码：「xxx 模块有什么功能？」「如何调用 yyy？」「zzz 的输入输出是什么？」系统会基于**代码图谱知识库**进行检索，再用大模型生成答案。

具体来说，我们把分析产出的**图谱数据**和**经验路径**，通过 `build_graph_index` 构建成向量索引。用户提问时，先做向量检索，再重排，最后把相关上下文交给 LLM 生成回答。

这样做的好处是：**答案有据可依**，不会凭空编造，而是来自真实的代码结构和分析结果。

---

**【画面】** 展示问答示例（可做简单 Demo）

**【旁白】**

例如，你可以问「项目的入口点有哪些」「某个类的主要职责是什么」「数据是怎么从 A 流到 B 的」。系统会从图谱和路径知识中检索相关内容，并给出清晰、有针对性的回答。

这就是 AI用：**代码仓库变成可检索的知识库，直接回答功能、用法、逻辑相关的问题。**

---

## 【第三部分：AI写 5:30–7:30】

**【画面】** 切换到「AI写」板块，展示多 Agent 架构

**【旁白】**

第三部分，**AI写**：代码生成系统。

不仅要「能问」，还要「能写」。但写代码不能盲目，必须**先理解项目**。

因此，我们采用了**多 Agent 协同**的架构：

- **Planner** 负责把用户目标拆解成可执行的步骤；
- **Coder** 根据步骤进行文件的读取、创建和修改；
- **Reviewer** 对改动进行审查；
- **ErrorSolver** 负责处理运行错误和回归问题；
- 而 **KnowledgeAgent** 则通过 RAG，在需要时为其他 Agent 提供代码仓库的知识。

---

**【画面】** 展示知识检索 → 计划 → 生成的流程图

**【旁白】**

流程大致是：用户提出目标 → KnowledgeAgent 从 RAG 中检索相关代码 → Planner 制定计划 → Coder 基于上下文生成或修改代码 → Reviewer 检查后输出。

这样，生成的代码会**与现有项目风格、结构保持一致**，而不是凭空创作。我们称之为**知识驱动的代码生成**。

总结 AI写：**不是盲目生成，而是先查知识库、再写代码，产出更贴合项目实际情况。**

## 【整体串联 7:30–8:30】

**【画面】** 端到端流程图：分析 → 索引 → 问答/生成

**【旁白】**

把三部分串起来看：

**第一步**，选择项目路径，运行代码分析。系统自动产出调用图、CFG、DFG、功能分区、经验路径等。

**第二步**，用 `build_graph_index` 把图谱和路径构建成向量索引。

**第三步**，两条路：

- **AI用**：通过 GraphRAG 直接回答用户关于功能、用法的问题；
- **AI写**：通过多 Agent 系统，基于检索到的知识生成或修改代码。

三者形成一个闭环：**AI学** 提供基础，**AI用** 和 **AI写** 在此基础上发挥作用。

## 【痛点回顾与总结 8:30–9:30】

**【画面】** 痛点 → 能力对照表

**【旁白】**

再回到开头的痛点，我们分别做了什么：

- **读不完** → 自动分析，多维度图谱，按功能组织，不再依赖人肉通读；
- **理不清** → 调用图、CFG、DFG、功能分区、经验路径，把结构说清楚；
- **找不到** → 功能分区、入口点、路径级分析，按功能快速定位；
- **问不了** → RAG 问答，直接回答功能、用法、逻辑；
- **改不动** → 知识驱动的代码生成，多 Agent 协同，产出与项目一致。

本质上，我们做的是：**把代码仓库变成 structured knowledge，让 AI 先学、再用、再写。**
