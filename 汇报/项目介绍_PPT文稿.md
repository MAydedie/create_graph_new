## 第1部分：痛点与价值主张

### 幻灯片 2：开发者面临的真实痛点

**标题**: 大型项目理解：开发者共同的难题

**内容要点**:
- 📚 **读不完**：动辄上万行代码，手动通读耗时数周
- 🕸️ **理不清**：类、方法、调用关系错综复杂，难以形成整体认知
- 🔍 **找不到**：想了解某个功能实现，不知道该从哪看起
- 📝 **问不了**：代码没有结构化知识，无法像文档一样检索问答
- ✍️ **改不动**：理解成本高，改代码怕牵一发动全身

**结论**: 传统方式下，大型项目理解主要靠「人肉读代码」，效率低、易出错、难传承。

---

### 幻灯片 3：我们的解决方案

**标题**: 三层能力：AI学 → AI用 → AI写

**内容要点**:
1. **AI学**：代码分析系统，自动把仓库变成 structured knowledge
2. **AI用**：RAG 问答，像问文档一样问代码
3. **AI写**：代码生成系统，基于仓库知识直接产出可用代码

**图示**: 
```
代码仓库 ──► [AI学] 分析 ──► 图谱+功能分区 ──► [AI用] RAG 问答
                                         └──► [AI写] 代码生成
```

## 第2部分：AI学 - 代码分析系统

### 幻灯片 4：以「方法」为核心的代码理解

**标题**: 方法级分析：符合人的思维习惯

**内容要点**:
- 代码的重要单元是 **方法/函数**，不是文件、不是行
- 我们围绕方法构建：调用关系、控制流、数据流、功能归属
- 支持：类、方法、函数、继承、跨文件调用
- 产出：符号表、调用图、CFG、DFG、超图

**一句话**: 让人用「功能」的视角理解代码，而不是「文件」的视角。

---

### 幻灯片 5：多种图：从结构到行为

**标题**: 多维度图谱：结构 + 行为 一目了然

**图类型**:
| 图类型 | 含义 | 用途 |
|--------|------|------|
| 调用图 | 方法间的调用关系 | 理解模块依赖、执行路径 |
| 超图 | 多对多调用关系 | 识别复杂调用模式 |
| 控制流图(CFG) | 方法内分支、循环 | 理解逻辑结构 |
| 数据流图(DFG) | 变量读写与流动 | 理解输入输出、副作用 |
| 分区图 | 按功能聚合的方法组 | 按「功能模块」浏览 |

**结论**: 不同粒度、不同维度，满足从宏观到微观的理解需求。

---

### 幻灯片 6：功能分析：按人的需求理解代码

**标题**: 功能分区与语义分析

**内容要点**:
- **功能分区**：LLM 结合项目结构，自动划分「核心业务」「工具辅助」「外部接口」等模块
- **经验路径（功能路径）**：从入口到关键逻辑的调用链，带语义描述、CFG、DFG
- **四层嵌套模型**：功能层 → 文件夹层 → 代码元素层 → 代码细节层
- **重要代码标注**：区分入口点、核心逻辑、工具类、支持性代码

**一句话**: 不是「文件树」，而是「功能树」，按人的认知组织代码知识。

---

### 幻灯片 7：可视化与交互

**标题**: 交互式图谱，所见即所得

**内容要点**:
- Cytoscape.js 交互式图谱：缩放、拖拽、搜索
- 节点类型：类(蓝)、方法(绿)、函数(红)
- 关系类型：调用、继承、包含
- 分区级：调用图、控制流、数据流、入口点
- 路径级：单条功能路径的 CFG/DFG/Mermaid

## 第3部分：AI用 - RAG 问答系统

### 幻灯片 8：像问文档一样问代码

**标题**: 基于代码图谱的 RAG 问答

**内容要点**:
- **问题示例**：「xxx 模块有什么功能？」「如何调用 yyy？」「zzz 的输入输出是什么？」
- **数据来源**：代码分析产出的 graph_data、experience_paths
- **流程**：build_graph_index 构建向量索引 → 用户提问 → 检索 → 重排 → LLM 生成答案
- **知识块**：类、方法、功能路径、CFG/DFG 等，统一向量化

**结论**: 代码仓库变成可检索的知识库，直接回答功能、用法、逻辑相关问题。

---

### 幻灯片 9：RAG 技术架构

**标题**: 检索增强生成

**流程**:
```
用户问题 ──► Embedding ──► FAISS 向量检索 ──► 重排(Reranker)
                                              │
                                              ▼
                                         上下文 + 问题
                                              │
                                              ▼
                                         LLM 生成答案
```

**特点**: 基于分析结果构建索引，问答有据可依，减少幻觉。

## 第4部分：AI写 - 代码生成系统

### 幻灯片 10：多 Agent 协同写代码

**标题**: 从用户目标到可执行代码

**内容要点**:
- **Orchestrator**：协调 Planner、Coder、Reviewer、ErrorSolver
- **Planner**：分解用户目标为可执行步骤
- **Coder**：根据步骤读写文件、生成/修改代码
- **Reviewer**：审查改动
- **ErrorSolver**：处理运行错误、回归问题
- **KnowledgeAgent**：调用 RAG，为各 Agent 提供代码仓库知识

**结论**: 不是盲目生成，而是「先查知识库、再写代码」，与仓库结构、风格一致。

---

### 幻灯片 11：知识驱动的代码生成

**标题**: 生成前先「懂」项目

**流程**:
```
用户目标 ──► KnowledgeAgent 检索相关代码 ──► Planner 制定计划
                                              │
                                              ▼
                                    Coder 基于上下文修改/生成
                                              │
                                              ▼
                                    Reviewer 检查 ──► 输出
```

**优势**: 生成内容与现有代码风格、结构保持一致，减少手动调整。
